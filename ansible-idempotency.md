## What Ansible does

Ansible is responsible for configuration -- the "what is installed and how is it set up" layer.
After Terraform creates the EC2 instance, Ansible connects to it over SSH and:

- Installs system packages (Java, Python, curl, and others)
- Installs and configures Payara
- Installs and configures Solr
- Installs and configures Apache with SSL
- Deploys the Dataverse WAR file into Payara
- Applies Dataverse configuration via the Dataverse API
- Sets JVM options in Payara for Dataverse to use

All of this is defined in the `dataverse-ansible` role -- a structured collection of tasks,
templates, handlers, and variable files.

## Role structure

The `dataverse-ansible` repo is an Ansible role. The key directories are:

```
dataverse-ansible/
  tasks/          <- what to do (ordered list of actions)
  handlers/       <- actions triggered by other tasks (e.g., restart Payara)
  templates/      <- config file templates with variable substitution
  defaults/       <- default variable values
  vars/           <- higher-priority variable values
  group_vars/     <- environment-specific overrides (not in the role itself, used by playbooks)
```

The `tasks/main.yml` file is the entry point. It calls out to other task files
(e.g., `tasks/payara.yml`, `tasks/solr.yml`) to keep things organized.

## Idempotency

An **idempotent** operation produces the same result whether you run it once or ten times.
For Ansible, this means: running the playbook on an already-configured server should make
no changes, because everything is already in the desired state.

Why this matters for operations:

- You can safely re-run the playbook after a partial failure without worrying about double-applying changes
- You can use `--check` mode to see what *would* change without actually changing anything
- A playbook that shows no changes is a signal that the server matches the expected configuration

Ansible achieves idempotency through modules that check state before acting:

```yaml
- name: Install Java
  apt:
    name: openjdk-11-jdk
    state: present   # "present" means "install if not already installed"
```

The `apt` module checks whether the package is installed before trying to install it.
If it is already installed, Ansible reports `ok` and moves on. If not, it installs and
reports `changed`.

::::::::::::::::::::::::::::::::::::: callout

### Not all tasks are idempotent by default

Tasks using `shell` or `command` modules run every time unless you add a `creates:` or
`when:` condition. The work to make `dataverse-ansible` reliably idempotent involved
identifying those tasks and adding the right conditions. When writing new tasks,
prefer modules over shell commands where one exists.

## group_vars and environment overrides

The `group_vars/` directory holds variable files that override defaults for specific
environments or host groups.

For example, the production environment might set:

```yaml
# group_vars/production.yml
dataverse_doi_provider: "EZID"
dataverse_use_test_cert: false
```

While the test environment sets:

```yaml
# group_vars/test.yml
dataverse_doi_provider: "FAKE"
dataverse_use_test_cert: true
```

The same playbook runs against both environments; the variables control which behavior
each environment gets. This is how we keep staging and production separate without
maintaining two copies of the role.

## The inventory

Ansible needs to know which host to connect to. The inventory file lists hosts and their
connection details. For this project, the inventory is generated by Terraform output:

```
[dataverse]
ec2-12-34-56-78.us-west-2.compute.amazonaws.com ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/dataverse-key.pem
```

The Makefile generates and uses this inventory automatically when you run `make rebuild`.

### Dry run a playbook

From `dataverse-infrastructure`, run:

```bash
make ansible-check ENV=tim
```

(This runs the playbook with `--check --diff` against Tim's environment.)

Read through the output and identify:

1. Which tasks report `changed` and which report `ok`?
2. For any `changed` tasks, what would Ansible change?
3. What does a task reporting `skipping` mean?

:::::::::::::::::::::::::::::::::: solution

`ok` means the system already matches the desired state -- no change needed.
`changed` means Ansible would modify something.
`skipping` means the task has a `when:` condition that evaluated to false -- the task
was intentionally not run for this environment or host.

A fully correct system should show mostly `ok` with zero `changed`.

::::::::::::::::::::::::::::::::::::: keypoints

- `dataverse-ansible` installs and configures Payara, Solr, Apache, and Dataverse.
- An idempotent playbook makes no changes when the system is already in the desired state.
- `group_vars` provides environment-specific values without duplicating the role.
- The Ansible inventory is generated from Terraform output -- the Makefile handles this.
- Use `--check` mode to preview what a playbook would change without running it.
